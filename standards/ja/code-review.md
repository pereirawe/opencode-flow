# コードレビューガイドライン

## 原則

- スタイルではなく正確性をレビュー（スタイルは自動化すべき）
- 建設的かつ具体的に
- 小さな提案とブロッカーを分ける
- 満足したら承認、そうでなければ変更を要求

## チェックリスト

- [ ] コードは正しく、エッジケースを処理している
- [ ] テストは変更を適切にカバーしている
- [ ] セキュリティやパフォーマンスの後退がない
- [ ] API 変更は文書化されている
- [ ] エラーパスが処理されている
- [ ] デッドコードやコメントアウトがない
- [ ] プロジェクト規約に従っている

## レビューフロー

1. Committer が初期レビューを実施
2. Publish Requester がドメインごとにシニアレビューアを割り当て
3. シニアレビューアが並行してレビュー
4. マージ前に全レビューアの承認が必要
