# Code-Review-Richtlinien

## Prinzipien

- Auf Korrektheit prüfen, nicht auf Stil (Stil sollte automatisiert sein)
- Konstruktiv und spezifisch sein
- Kleine Vorschläge von blockierenden Problemen trennen
- Zustimmen wenn zufrieden, Änderungen anfordern wenn nicht

## Checkliste

- [ ] Code ist korrekt und behandelt Randfälle
- [ ] Tests decken die Änderung angemessen ab
- [ ] Keine Sicherheits- oder Performance-Regressionen
- [ ] API-Änderungen sind dokumentiert
- [ ] Fehlerpfade werden behandelt
- [ ] Kein toter oder auskommentierter Code
- [ ] Projektkonventionen werden eingehalten

## Review-Ablauf

1. Committer führt Erstprüfung durch
2. Publish Requester weist Senior Reviewers nach Domäne zu
3. Senior Reviewers prüfen parallel
4. Alle Prüfer müssen vor dem Merge zustimmen
