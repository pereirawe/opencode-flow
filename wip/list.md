1. O projeto deve ter varios agents
2. Cada agent deve ter uma função específica
3. Os agents devem ser capazes de se comunicar entre si
4. Os agents devem ser capazes de aprender e se adaptar ao ambiente
5. Os agents devem ser capazes de tomar decisões autônomas
6. Os agents devem ser capazes de colaborar para alcançar objetivos comuns
7. Os agents devem ser capazes de lidar com situações imprevistas
8. Os agents devem ser capazes de se auto-organizar e se auto-gerenciar
9. Os agents devem ser capazes de se auto-reparar e se auto-melhorar
10. Os agents devem ser capazes de se auto-proteger e se auto-defender
11. Os agents devem ser capazes de se auto-replicar e se auto-evoluir
12. Lista de agentes:
- Agente de CTO: responsável por liderar a equipe de tecnologia e garantir que os objetivos técnicos sejam alcançados.
- Agente de Gerente de Projeto: responsável por coordenar as atividades da equipe e garantir
- Agente de Product Owner: responsável por definir as prioridades do produto e garantir que as necessidades dos usuários sejam atendidas.
- Agente de Analista de Qualidade: responsável por garantir que o produto atenda aos padrões de qualidade e seja testado adequadamente.
- Agente de Desenvolvedor: responsável por escrever código e implementar as funcionalidades do produto.
- Agente de Commiter: responsável por revisar e aprovar as mudanças de código antes de serem integradas ao projeto.
- Agente de Publish Requester: responsável por criar solicitações de Merge Request para publicar as mudanças no repositório.
- Agente de Documentação: responsável por criar e manter a documentação do projeto.
- Agentes de Desenvolvedores Seniors Reviewers: responsáveis por revisar o código dos desenvolvedores e fornecer feedback para garantir a qualidade do código.
  - Desenvolvedor Senior Devops: responsável por revisar o código relacionado a infraestrutura e operações.
  - Desenvolvedor Senior Backend: responsável por revisar o código relacionado ao backend do sistema.
  - Desenvolvedor Senior Frontend: responsável por revisar o código relacionado ao frontend do sistema.
  - Desenvolvedor Senior Data: responsável por revisar o código relacionado a dados e banco de dados do sistema.
  - Desenvolvedor Senior Segurança: responsável por revisar o código relacionado à segurança do sistema.
  - Desenvolvedor Senior Performance: responsável por revisar o código relacionado à performance do sistema.
  - Desenvolvedor Senior UX/UI: responsável por revisar o código relacionado à experiência do usuário e interface do sistema.
  - Desenvolvedor Senior Runtime: responsável por revisar o código relacionado ao ambiente de execução do sistema.
  - Desenvolvedor Senior Mobile: responsável por revisar o código relacionado ao desenvolvimento mobile do sistema.
  - Desenvolvedor Senior QA: responsável por revisar o código relacionado a testes e qualidade do sistema.
  - Agente de Testes Automatizados: responsável por criar e manter os testes automatizados para garantir a qualidade do código.

13. o Fluxo de trabalho deve ser o seguinte:
1. O CTO define a visão técnica do projeto e estabelece as diretrizes para a equipe de tecnologia.
2. O Product Owner define as prioridades do produto e cria as histórias de usuário.
3. O Gerente de Projeto coordena as atividades da equipe e garante que as histórias de usuário sejam atribuídas aos desenvolvedores.
3. O Analista de Qualidade trabalha com o Product Owner para garantir que as histórias de usuário sejam testáveis e atendam aos padrões de qualidade. E os testes automatizados são criados para garantir a qualidade do código. 
4. (Quem deveria fazer este perfil) Os Criterios de Aceitação são definidos para garantir que as histórias de usuário sejam testáveis e atendam aos padrões de qualidade.
5. O Tech Lead fornece orientação técnica e suporte aos desenvolvedores durante o processo de desenvolvimento. Ele define as melhores práticas de codificação e garante que o código seja de alta qualidade e siga as diretrizes estabelecidas pelo CTO..
6. Os Desenvolvedores escrevem o código para implementar as funcionalidades do produto. Eles também criam testes automatizados para garantir a qualidade do código.
7. Os Commiter revisam o código dos desenvolvedores e aprovam as mudanças antes de serem integradas ao projeto.
8. O Publish Requester cria solicitações de Merge Request para publicar as mudanças no repositório.
9. Os Analistas de Qualidade garantem que o produto atenda aos padrões de qualidade e seja testado adequadamente.
10. Os Testes Automatizados são criados e mantidos para garantir a qualidade do código.
11. Os Desenvolvedores Seniors Reviewers revisam o código dos desenvolvedores e fornecem feedback para garantir a qualidade do código.

14. Um dos agentes ou skills deve estar focado na criação e manutenção de um arquivo de issues conhecido, onde serão listados os problemas conhecidos do projeto, suas soluções e status de resolução. Este arquivo deve ser atualizado regularmente para garantir que a equipe esteja ciente dos problemas atuais e possa trabalhar para resolvê-los.
15. Commandos podem criar issues, promover para resolver ou desenvolver issues existentes, ou marcar como resolvidas. O arquivo de issues conhecido deve ser a fonte de verdade para o trabalho rastreado e deve ser atualizado regularmente para refletir o status atual dos problemas conhecidos do projeto.
16. O agente ou skill responsável pela criação e manutenção do arquivo de issues conhecido deve ser capaz de escanear o código em busca de bugs, problemas de segurança, problemas de concorrência e design drift, e atualizar as entradas existentes antes de criar novas.
17. Agentes ou skills devem ter as suas funções bem definidas e documentadas. Com comandos claros para cada função, e um processo de comunicação eficiente entre eles para garantir que as tarefas sejam realizadas de forma coordenada e eficiente. A documentação deve incluir detalhes sobre as responsabilidades de cada agente ou skill, os processos de trabalho, e as diretrizes para a colaboração entre eles.
18. Todos os agents devem estar limpos de contextos específicos, sem conteúdo de linguagem de programação, ou detalhes técnicos específicos, para garantir que possam se adaptar a diferentes projetos e ambientes de desenvolvimento. Eles devem ser projetados para serem flexíveis e capazes de lidar com uma variedade de situações e desafios que possam surgir durante o desenvolvimento do projeto.
19. Deve ter uma pasta para a documentação de padrão de PR, de Issues, de Commits, de Branches, e outros padrões de desenvolvimento que possam ser relevantes para o projeto. Esta documentação deve ser facilmente acessível para toda a equipe e deve ser atualizada regularmente para refletir as melhores práticas e diretrizes do projeto.
